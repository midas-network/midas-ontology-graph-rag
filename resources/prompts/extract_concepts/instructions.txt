LLM Extraction Prompt v4 — Constrained JSON Mode (Schema + Controlled Vocabulary)

TASK
Extract structured metadata from the study TITLE and ABSTRACT.

IMPORTANT CONTEXT
- Output is schema-constrained JSON (the schema is the source of truth).
- Use the controlled MIDAS vocabulary file for preferred terms where applicable.
- Use the MIDAS synonym map to normalize common variants to preferred terms.

PRIORITY ORDER (when deciding what to output)
1) Constrained schema (required fields, JSON shape, enum values, exact enum spelling)
2) Controlled MIDAS vocabulary (preferred terms)
3) MIDAS synonym map (normalize into preferred terms)
4) Title + abstract evidence
5) These prompt instructions / field notes

CRITICAL RULES
- Use evidence from BOTH title and abstract.
- Prefer explicit evidence over inference.
- Do not use external knowledge except high-confidence normalization (e.g., disease/pathogen naming).
- If the abstract/title does not support a specific value, use the appropriate schema-compatible absent value for that field.
- Do not invent enum values. Use exact schema spellings only.
- Do not invent controlled-vocabulary labels. Prefer MIDAS terms for controlled fields.

HARD OUTPUT CONTRACT (NON-NEGOTIABLE)
- Output MUST be a JSON object.
- The first character MUST be { and the last character MUST be }.
- No leading text, no trailing text, no surrounding quotes, no markdown, no code fences.
- Do not return JSON as a string.

CONTROLLED VOCABULARY FIELDS
For these fields, prefer exact MIDAS ontology terms from the vocab list:
- data_used
- intervention_types
- model_type

If a candidate value is a synonym/variant:
- normalize it using the synonym map to the preferred MIDAS term.

If no MIDAS term matches:
- free text is allowed for those fields, but keep it concise and canonical.

EVIDENCE POLICY (STRICT)
Allowed:
- High-confidence normalization (disease/pathogen naming)
- Minimal domain inference when strongly justified by explicit wording

Not allowed:
- Guessing code availability
- Guessing calibration methods
- Guessing data sources/providers
- Adding likely populations/settings/geographies not supported by title/abstract

If unsure:
- choose the schema-compatible absent value for that field
- or use [] for list fields with no supported values

MULTI-VALUE RULES
- Include multiple values only when distinctly supported.
- Deduplicate near-duplicates (e.g., US/USA/United States -> one canonical form).
- Avoid paraphrase duplicates if a controlled MIDAS term exists.

REASONING FIELD RULES
- Keep reasoning brief and evidence-based.
- Explain the evidence or normalization choice.
- Use "[Inferred]" only when a small, high-confidence inference was necessary.

CALIBRATION VS VALIDATION (IMPORTANT)
calibration_mentioned = yes ONLY for fitting / estimation / data assimilation methods.
Examples that count as calibration techniques:
- MCMC
- maximum likelihood estimation
- least squares
- particle filter
- data assimilation
- Bayesian inference
- ABC

Do NOT treat evaluation metrics as calibration techniques:
- WIS
- MAE
- RMSE
- forecast accuracy summaries
- coverage metrics

FINAL MENTAL CHECK
- Did I use exact schema enum spellings?
- Did I prefer MIDAS controlled vocab for data_used/intervention_types/model_type?
- Did I normalize synonyms to preferred MIDAS terms?
- Did I avoid unsupported inference?
- Did I use [] for empty list fields?
- Did I use a schema-compatible absent value for enum/single-value fields when needed?
